### Урок 7.1 — Что такое миграции
**Экраны:** теория → квиз

#### 📖 Теория
Проект уже работает, в базе лежат данные. Тебе нужно добавить колонку `status` в таблицу `payments`. Нельзя просто удалить таблицу и создать заново — потеряются данные.

**Миграция** — это пошаговое, версионируемое изменение структуры БД. Каждое изменение — отдельный файл с двумя инструкциями:
- **upgrade** — как применить изменение (добавить колонку);
- **downgrade** — как откатить (убрать колонку).

Аналогия: миграции — это система контроля версий, но для структуры базы. Как git-коммиты, только про таблицы.

🐍 Пай: «Миграция — это коммит для базы данных. У неё есть „вперёд“ (upgrade) и „назад“ (downgrade)».

#### ❓ Квиз
Зачем нужны миграции вместо «удалить таблицу и создать заново»?
- ✅ `Чтобы менять структуру базы, не теряя уже накопленные данные`
- `Чтобы ускорить запросы` — миграции про структуру, а не про скорость.
- `Чтобы не писать SQL` — SQL никуда не девается, миграции его упорядочивают.
- `Чтобы шифровать данные` — миграции к шифрованию не относятся.

### Урок 7.2 — Команды Alembic
**Экраны:** теория → пазл

#### 📖 Теория
**Alembic** — инструмент миграций для SQLAlchemy. Основные команды:

```bash
alembic init alembic                          # один раз: создать папку миграций
alembic revision -m "add status to payments"  # создать пустую миграцию
alembic revision --autogenerate -m "..."       # создать по разнице моделей и БД
alembic upgrade head                           # применить все новые миграции
alembic downgrade -1                           # откатить на одну назад
```

`--autogenerate` сравнивает твои модели с текущей базой и сам пишет, что изменилось. Это удобно, но результат всегда нужно просматривать глазами.

🐍 Пай: «`upgrade head` — накатить всё до последней версии. `downgrade -1` — шаг назад. Как перемотка истории базы».

**Строки:** `изменить модель в коде` · `alembic revision --autogenerate` · `проверить файл миграции глазами` · `alembic upgrade head`
**Правильный порядок:** `изменить модель в коде` · `alembic revision --autogenerate` · `проверить файл миграции глазами` · `alembic upgrade head`

### Урок 7.3 — Содержимое миграции
**Экраны:** теория → квиз → задача

#### 📖 Теория
Внутри файла миграции — две функции:

```python
def upgrade():
    op.add_column("payments", sa.Column("status", sa.String(), nullable=False, server_default="created"))

def downgrade():
    op.drop_column("payments", "status")
```

Правило: `downgrade` должен **отменять ровно то**, что сделал `upgrade`. Добавил колонку — в downgrade удали её. Так миграцию можно безопасно откатить.

🐍 Пай: «upgrade и downgrade — зеркало. Что одна добавила, другая убирает. Тогда история обратима».

#### ❓ Квиз
Что должна делать функция `downgrade`, если `upgrade` добавила колонку?
- ✅ `Удалять эту же колонку — чтобы откат вернул схему как было`
- `То же, что upgrade` — тогда откат ничего не отменит.
- `Удалять всю таблицу` — это лишнее, отменяем только своё изменение.
- `Ничего, downgrade не нужен` — без него миграцию нельзя откатить.

#### 💻 Задача
**Условие (самопроверка):** напиши пару `upgrade`/`downgrade` для добавления колонки `external_id` (строка) в таблицу `payments`.
**Стартовый код:**
```python
def upgrade():
    # добавь колонку external_id
    pass

def downgrade():
    # отмени добавление
    pass
```
**Подсказки:**
1. Добавление — `op.add_column("payments", sa.Column("external_id", sa.String()))`.
2. Откат — `op.drop_column("payments", "external_id")`.
3. Решение:
```python
def upgrade():
    op.add_column("payments", sa.Column("external_id", sa.String()))

def downgrade():
    op.drop_column("payments", "external_id")
```
**Автотесты:** самопроверка: upgrade добавляет external_id, downgrade его удаляет.

### Урок 7.4 — Типичные ошибки
**Экраны:** теория → квиз

#### 📖 Теория
**Ошибки новичков с миграциями:**
- забыть написать `downgrade` — тогда откат невозможен;
- применить `--autogenerate` и не посмотреть результат — он может удалить нужную колонку;
- добавить колонку `NOT NULL` без `server_default` в непустую таблицу — старые строки нарушат правило, миграция упадёт;
- править уже применённую на проде миграцию вместо создания новой.

Главное правило прода: **новое изменение = новая миграция**, а не правка старой.

🐍 Пай: «Не правь историю, которую уже накатили на прод. Делай новую миграцию поверх».

#### ❓ Квиз
Ты добавляешь колонку `NOT NULL` в таблицу, где уже есть строки. Что сделать, чтобы миграция не упала?
- ✅ `Задать значение по умолчанию (server_default) для существующих строк`
- `Удалить все строки` — это потеря данных, так нельзя.
- `Сделать колонку primary key` — это не решает проблему пустых значений.
- `Игнорировать ошибку` — миграция просто не применится.
