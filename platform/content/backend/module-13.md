### Урок 13.1 — Зачем нужен Docker
**Экраны:** теория → квиз

#### 📖 Теория
Классическая беда: «у меня на ноутбуке работает, а на сервере — нет». Причина — разные версии Python, библиотек, настроек.

**Docker** решает это: он упаковывает приложение вместе со всем окружением в **контейнер**, который одинаково запускается где угодно — на твоём ноутбуке, у коллеги, на проде.

Аналогия: Docker — это транспортный контейнер. Что внутри упаковано, то и приедет в порт назначения без сюрпризов.

🐍 Пай: «Docker убивает фразу „у меня работает“. Работает в контейнере — работает у всех».

#### ❓ Квиз
Какую проблему решает Docker?
- ✅ `«На моей машине работает, а на сервере нет» — за счёт одинакового окружения везде`
- `Медленный интернет` — Docker не про сеть.
- `Нехватку памяти` — это не его задача.
- `Ошибки в коде` — Docker не исправляет баги, он про окружение.

### Урок 13.2 — Образ и контейнер
**Экраны:** теория → квиз

#### 📖 Теория
Два главных слова, которые путают:
- **Image (образ)** — застывший слепок: код + окружение. Как установочный диск или рецепт.
- **Container (контейнер)** — запущенный экземпляр образа. Как блюдо, приготовленное по рецепту.

Из одного образа можно запустить много контейнеров. Образ — шаблон, контейнер — живой запущенный процесс.

```text
Dockerfile  →  (build)  →  Image  →  (run)  →  Container
  рецепт                   слепок            живой запуск
```

🐍 Пай: «Образ — рецепт. Контейнер — готовое блюдо. Один рецепт — сколько угодно порций».

#### ❓ Квиз
В чём разница между образом (image) и контейнером (container)?
- ✅ `Образ — застывший шаблон, контейнер — запущенный из него экземпляр`
- `Это одно и то же` — нет, шаблон и запуск — разные вещи.
- `Контейнер — это рецепт` — рецепт это образ, а контейнер — готовый запуск.
- `Образ запускается, контейнер хранится` — наоборот: запускается контейнер.

### Урок 13.3 — Dockerfile
**Экраны:** теория → пазл

#### 📖 Теория
**Dockerfile** — рецепт сборки образа, шаг за шагом:

```dockerfile
FROM python:3.12-slim          # базовый образ с Python
WORKDIR /app                   # рабочая папка внутри контейнера
COPY requirements.txt .        # копируем список зависимостей
RUN pip install -r requirements.txt   # ставим зависимости
COPY . .                       # копируем остальной код
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0"]   # команда запуска
```

Порядок важен: зависимости копируют и ставят **до** остального кода. Тогда при правке кода Docker не переустанавливает библиотеки заново — он переиспользует кэш слоёв. Это сильно ускоряет сборку.

🐍 Пай: «Сначала requirements и install, потом код. Так Docker кэширует тяжёлый install и не повторяет его на каждую правку».

**Правильный порядок:** `FROM python:3.12-slim` · `WORKDIR /app` · `COPY requirements.txt .` · `RUN pip install -r requirements.txt` · `COPY . .` · `CMD ["uvicorn", "app.main:app"]`

### Урок 13.4 — docker-compose
**Экраны:** теория → баг

#### 📖 Теория
Сервису нужен не только сам app, но и база, и Redis. Запускать их по одному вручную — мучение. **docker-compose** описывает все сервисы в одном файле и поднимает их вместе одной командой:

```yaml
services:
  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      DATABASE_URL: postgresql://user:pass@db:5432/app
    depends_on:
      - db
  db:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data
volumes:
  pgdata:
```

Команды:
```bash
docker compose up --build   # собрать и запустить всё
docker compose down         # остановить
```

Важное: внутри compose сервисы видят друг друга по имени — app обращается к базе по хосту `db`, а не `localhost`.

🐍 Пай: «compose — это дирижёр. Одна команда — и app, база, Redis поднимаются вместе и слышат друг друга по именам».

#### 🐞 Найди баг
**Код:**
```yaml
# app внутри docker-compose подключается к базе так:
environment:
  DATABASE_URL: postgresql://user:pass@localhost:5432/app
```
**Ошибка:** строка 3 — внутри compose база доступна не по `localhost`, а по имени сервиса `db`.
**Правильно:** `DATABASE_URL: postgresql://user:pass@db:5432/app`
**Объяснение после решения:** для контейнера `localhost` — это он сам. Чтобы достучаться до базы, используют имя её сервиса из compose (`db`).

### Урок 13.5 — Volumes и переменные окружения
**Экраны:** теория → задача → квиз

#### 📖 Теория
- **Volume (том)** — постоянное хранилище. Контейнер можно удалить и пересоздать, а данные в volume останутся. Без volume база потеряет всё при перезапуске.
- **Environment (переменные окружения)** — настройки и секреты, которые передают контейнеру снаружи (адрес базы, ключи). В коде их не пишут — берут из окружения.

```text
volume        → данные переживают перезапуск контейнера
environment   → секреты и настройки приходят снаружи, не из кода
```

🐍 Пай: «Volume — сейф для данных, переживает контейнер. Environment — конверт с настройками, передают снаружи».

#### 💻 Задача
**Условие:** из переменной `DATABASE_URL` достань хост базы. На вход — строка вида `postgresql://user:pass@db:5432/app`. Выведи хост (часть между `@` и `:`) — для примера это `db`.
**Стартовый код:**
```python
url = input()  # postgresql://user:pass@db:5432/app
# выведи хост базы
```
**Подсказки:**
1. Часть после `@` — это `db:5432/app`. Возьми её: `url.split("@")[1]`.
2. Хост — до двоеточия: ещё раз `.split(":")[0]`.
3. Решение:
```python
url = input()
host = url.split("@")[1].split(":")[0]
print(host)
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `postgresql://user:pass@db:5432/app` | `db` |
| `postgresql://admin:secret@localhost:5432/payments` | `localhost` |

#### ❓ Квиз
Зачем базе данных в Docker нужен volume?
- ✅ `Чтобы данные сохранялись, даже если контейнер удалить и пересоздать`
- `Чтобы ускорить запросы` — volume про сохранность, не про скорость.
- `Чтобы спрятать пароли` — секреты прячут в environment, а не в volume.
- `Чтобы открыть порт` — порты — это другая настройка.
