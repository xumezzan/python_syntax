### Урок 8.1 — Authentication и Authorization
**Экраны:** теория → квиз

#### 📖 Теория
Два похожих слова, которые путают. Разведём их раз и навсегда:
- **Authentication (аутентификация)** — «кто ты?». Проверка личности: логин + пароль, токен.
- **Authorization (авторизация)** — «что тебе можно?». Проверка прав: этому пользователю разрешено удалять платежи или нет.

Аналогия с офисом: аутентификация — охранник проверил твой пропуск (ты — это ты). Авторизация — твой пропуск открывает только нужные двери (не везде можно).

Сначала всегда authentication, потом authorization. Сначала узнаём кто, потом решаем что можно.

🐍 Пай: «Authentication — кто ты. Authorization — что тебе можно. Сначала пропуск, потом двери».

#### ❓ Квиз
Пользователь вошёл с правильным паролем, но пытается удалить чужой платёж, и система не даёт. Что сработало?
- ✅ `Authorization — личность подтверждена, но прав на это действие нет`
- `Authentication` — она уже прошла успешно (пароль верный).
- `Хэширование` — это про хранение паролей, не про доступ.
- `Миграция` — вообще про структуру БД.

### Урок 8.2 — Хэширование паролей
**Экраны:** теория → задача → баг

#### 📖 Теория
Пароли **никогда** не хранят в открытом виде. Хранят их **хэш** — необратимый «отпечаток». При входе хэшируют введённый пароль и сравнивают с сохранённым.

```python
import hashlib
hash1 = hashlib.sha256("secret123".encode()).hexdigest()
```

Один и тот же пароль всегда даёт один и тот же хэш, но из хэша пароль восстановить нельзя. В реальных проектах берут специальные алгоритмы для паролей (`bcrypt`, `argon2`) — они медленные нарочно, чтобы их было трудно подобрать.

**Ошибка новичка:** сохранить пароль как есть в колонку `password`. Если базу украдут — утекут все пароли. Только хэш.

🐍 Пай: «Пароль — это секрет даже от тебя самого. Храни отпечаток, а не оригинал».

#### 💻 Задача
**Условие:** «зарегистрируй» пользователя — выведи sha256-хэш его пароля. Считай пароль из ввода и выведи `hashlib.sha256(...).hexdigest()`.
**Стартовый код:**
```python
import hashlib
password = input()
# выведи sha256-хэш пароля
```
**Подсказки:**
1. Строку нужно превратить в байты: `password.encode()`.
2. `hashlib.sha256(password.encode()).hexdigest()`.
3. Решение:
```python
import hashlib
password = input()
print(hashlib.sha256(password.encode()).hexdigest())
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `secret123` | `fcf730b6d95236ecd3c9fc2d92d7b6b2bb061514961aec041d6c7a7192f592e4` |

#### 🐞 Найди баг
**Код:**
```python
# регистрация пользователя
user = {"email": "ivan@mail.uz", "password": "secret123"}
db.save(user)
```
**Ошибка:** строка 2 — пароль сохраняется в открытом виде; при утечке базы он скомпрометирован.
**Правильно:** `user = {"email": "ivan@mail.uz", "hashed_password": hashlib.sha256("secret123".encode()).hexdigest()}`
**Объяснение после решения:** в базу кладут только хэш пароля, а не сам пароль. Поле обычно так и называют — `hashed_password`.

### Урок 8.3 — JWT-токен
**Экраны:** теория → задача

#### 📖 Теория
После входа сервер не хочет проверять пароль на каждом запросе. Он выдаёт **JWT-токен** — подписанный «пропуск», который клиент прикладывает к каждому запросу.

JWT состоит из трёх частей через точку:

```text
header . payload . signature
```

- **header** — алгоритм подписи;
- **payload** — данные: кто пользователь, его роль, срок годности;
- **signature** — подпись сервера: если токен подделать, подпись не сойдётся.

Клиент шлёт токен в заголовке: `Authorization: Bearer <token>`. Сервер проверяет подпись и читает payload.

🐍 Пай: «JWT — это пропуск с печатью. Подделать payload можно, но без секретного ключа печать (подпись) не подделать».

#### 💻 Задача
**Условие:** прочитай роль из payload JWT-токена. Тебе на вход приходит токен вида `header.payload.signature`, где payload — это base64url от JSON. Декодируй payload и выведи значение `role`.
**Стартовый код:**
```python
import base64, json
token = input()
payload_b64 = token.split(".")[1]
payload_b64 += "=" * (-len(payload_b64) % 4)  # добиваем padding
# декодируй payload и выведи роль
```
**Подсказки:**
1. Декодируй: `base64.urlsafe_b64decode(payload_b64)`.
2. Преврати в словарь через `json.loads`, достань `role`.
3. Решение:
```python
import base64, json
token = input()
payload_b64 = token.split(".")[1]
payload_b64 += "=" * (-len(payload_b64) % 4)
data = json.loads(base64.urlsafe_b64decode(payload_b64))
print(data["role"])
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJpdmFuIiwicm9sZSI6ImFkbWluIn0.sig` | `admin` |
| `eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJhbm5hIiwicm9sZSI6InZpZXdlciJ9.sig` | `viewer` |

### Урок 8.4 — Роли и права
**Экраны:** теория → задача → квиз

#### 📖 Теория
Чтобы не раздавать права каждому действию вручную, пользователям дают **роли**, а ролям — набор разрешённых действий:

```text
admin     — может всё
manager   — создавать и просматривать платежи
developer — читать логи и платежи
viewer    — только смотреть
```

Закрытый эндпоинт проверяет: «есть ли у роли этого пользователя нужное право?». Нет права — `403 Forbidden`.

🐍 Пай: «Роль — это набор ключей. admin носит связку от всех дверей, viewer — только от смотровой».

#### 💻 Задача
**Условие:** проверь право на действие. На вход — две строки: роль и действие. Разреши действие `delete` только роли `admin`; чтение (`read`) разрешено всем ролям. Выведи `allowed` или `forbidden`.
**Стартовый код:**
```python
role = input()
action = input()
# выведи allowed или forbidden
```
**Подсказки:**
1. `read` всегда `allowed`.
2. `delete` — `allowed` только если `role == "admin"`, иначе `forbidden`.
3. Решение:
```python
role = input()
action = input()
if action == "read":
    print("allowed")
elif action == "delete" and role == "admin":
    print("allowed")
else:
    print("forbidden")
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `admin` ⏎ `delete` | `allowed` |
| `viewer` ⏎ `delete` | `forbidden` |
| `viewer` ⏎ `read` | `allowed` |

#### ❓ Квиз
Пользователь с ролью `viewer` пытается удалить платёж. Какой статус-код вернёт API?
- ✅ `403 Forbidden — личность известна, но прав на действие нет`
- `401 Unauthorized` — это когда вообще не вошёл или токен плохой.
- `404 Not Found` — платёж-то существует, дело в правах.
- `200 OK` — нет, действие запрещено, успехом тут не пахнет.

### Урок 8.5 — Защита эндпоинтов
**Экраны:** теория → квиз

#### 📖 Теория
В FastAPI закрытый эндпоинт требует токен через зависимость:

```python
@app.get("/payments")
def list_payments(user = Depends(get_current_user)):
    return ...
```

`Depends(get_current_user)` — это «прежде чем пустить, проверь токен». Нет токена или он плохой → `401`, до тела функции дело не дойдёт.

Различай два отказа:
- **401 Unauthorized** — ты вообще не представился (нет/плохой токен);
- **403 Forbidden** — представился, но прав не хватает.

🐍 Пай: «401 — „ты кто?“. 403 — „тебе сюда нельзя“. Разные двери, разные коды».

#### ❓ Квиз
Запрос к `/payments` пришёл вообще без заголовка Authorization. Что вернёт защищённый эндпоинт?
- ✅ `401 Unauthorized — токена нет, личность не подтверждена`
- `403 Forbidden` — это для случая «вошёл, но нет прав».
- `200 OK` — закрытый эндпоинт без токена не отдаёт данные.
- `500 Server Error` — это не сбой сервера, а отсутствие токена.

### Урок 8.6 — 🏆 БОСС: матрица доступа
**Экраны:** теория → задача

#### 📖 Теория
Собери проверку прав по таблице доступа. У каждой роли — свой набор разрешённых действий. Эндпоинт смотрит в эту матрицу и пускает или нет.

🐍 Пай: «Матрица „роль → действия“ — сердце авторизации. Один словарь решает, кому что можно».

#### 💻 Задача
**Условие:** есть матрица прав. На вход — роль и действие. Выведи `allowed`, если действие входит в список роли, иначе `forbidden`.
**Стартовый код:**
```python
permissions = {
    "admin": ["read", "create", "delete"],
    "manager": ["read", "create"],
    "viewer": ["read"],
}
role = input()
action = input()
# выведи allowed или forbidden по матрице
```
**Подсказки:**
1. Достань список прав роли: `permissions.get(role, [])`.
2. Проверь `action in ...`.
3. Решение:
```python
allowed = action in permissions.get(role, [])
print("allowed" if allowed else "forbidden")
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `manager` ⏎ `create` | `allowed` |
| `manager` ⏎ `delete` | `forbidden` |
| `viewer` ⏎ `read` | `allowed` |
| `guest` ⏎ `read` | `forbidden` |
