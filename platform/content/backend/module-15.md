### Урок 15.1 — Что такое logging
**Экраны:** теория → квиз

#### 📖 Теория
**Логирование** — это запись того, что происходит в программе, пока она работает. Без логов работающий сервис — чёрный ящик: что-то пошло не так, а почему — неизвестно.

`print` для этого не годится: его не отключить, нельзя задать важность, он не пишет время. В Python для логов есть модуль `logging` с **уровнями важности**:

| Уровень | Когда |
|---------|-------|
| `DEBUG` | мелкие детали для отладки |
| `INFO` | обычные события: «платёж создан» |
| `WARNING` | подозрительно, но не сломалось |
| `ERROR` | что-то упало |

🐍 Пай: «print — это крик в пустоту. Логи — это дневник сервиса, с временем, уровнем и смыслом».

#### ❓ Квиз
Чем логирование лучше обычного `print` в боевом сервисе?
- ✅ `У логов есть уровни важности, время и их можно фильтровать и отключать`
- `Ничем, это одно и то же` — нет, у логов больше возможностей.
- `print быстрее` — скорость тут не главное.
- `Логи нельзя читать` — наоборот, их и читают при разборе проблем.

### Урок 15.2 — Уровни логов
**Экраны:** теория → задача

#### 📖 Теория
У логов есть порог. Если выставить уровень `WARNING`, то `DEBUG` и `INFO` не покажутся — только `WARNING` и `ERROR`. Так в проде убирают шум и оставляют важное.

Важность по возрастанию:

```text
DEBUG (10) < INFO (20) < WARNING (30) < ERROR (40)
```

Сообщение проходит, если его уровень **не ниже** порога.

🐍 Пай: «Порог логов — как фильтр громкости. Поставил WARNING — мелочь молчит, слышно только важное».

#### 💻 Задача
**Условие:** есть порог логирования и уровень сообщения. На вход — две строки: порог и уровень (`DEBUG`/`INFO`/`WARNING`/`ERROR`). Выведи `show`, если уровень не ниже порога, иначе `skip`.
**Стартовый код:**
```python
levels = {"DEBUG": 10, "INFO": 20, "WARNING": 30, "ERROR": 40}
threshold = input()
message_level = input()
# выведи show или skip
```
**Подсказки:**
1. Сравни числовые веса: `levels[message_level] >= levels[threshold]`.
2. Если да — `show`, иначе `skip`.
3. Решение:
```python
levels = {"DEBUG": 10, "INFO": 20, "WARNING": 30, "ERROR": 40}
threshold = input()
message_level = input()
if levels[message_level] >= levels[threshold]:
    print("show")
else:
    print("skip")
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `WARNING` ⏎ `ERROR` | `show` |
| `WARNING` ⏎ `INFO` | `skip` |
| `DEBUG` ⏎ `DEBUG` | `show` |

### Урок 15.3 — request_id и trace_id
**Экраны:** теория → задача

#### 📖 Теория
Когда в логах тысячи строк от сотен запросов, как понять, какие из них относятся к одному платежу? Каждому запросу присваивают **request_id** — уникальную метку, и пишут её в каждый лог этого запроса.

**trace_id** — то же самое, но через несколько систем: SAP → backend → банк. По одному trace_id можно собрать весь путь платежа через все сервисы.

```text
[req-7] платёж создан
[req-7] отправлен в банк
[req-7] банк ответил success
```

По `req-7` мгновенно видно всю историю одного запроса.

🐍 Пай: «request_id — это номер заказа в логах. Один запрос — одна метка — и весь его путь как на ладони».

#### 💻 Задача
**Условие:** добавь request_id к сообщению лога. На вход — две строки: request_id и текст сообщения. Выведи в формате `[request_id] сообщение`.
**Стартовый код:**
```python
request_id = input()
message = input()
# выведи: [request_id] сообщение
```
**Подсказки:**
1. Используй f-строку: `f"[{request_id}] {message}"`.
2. Скобки — часть формата.
3. Решение:
```python
request_id = input()
message = input()
print(f"[{request_id}] {message}")
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `req-7` ⏎ `payment created` | `[req-7] payment created` |
| `req-12` ⏎ `sent to bank` | `[req-12] sent to bank` |

### Урок 15.4 — Структурные логи
**Экраны:** теория → задача → квиз

#### 📖 Теория
Текстовый лог `платёж 9 создан` человек прочитает, а программа-анализатор — с трудом. Поэтому в проде логи делают **структурными** — в JSON:

```python
import json, logging
log = {"level": "INFO", "event": "payment_created", "payment_id": 9, "request_id": "req-7"}
print(json.dumps(log))
```

Такие логи легко искать и фильтровать в системах мониторинга: «покажи все `event=payment_failed` за час». Машина читает поля, а не разбирает текст.

🐍 Пай: «Структурный лог — это лог-таблица, а не лог-сочинение. Машина по нему ищет за миллисекунды».

#### 💻 Задача
**Условие:** собери структурный лог события платежа. На вход — две строки: id платежа (число) и событие. Поля: `level`=`INFO`, `event`=событие, `payment_id`=число. Выведи JSON (`ensure_ascii=False`).
**Стартовый код:**
```python
import json
payment_id = int(input())
event = input()
# собери структурный лог и выведи как JSON
```
**Подсказки:**
1. Создай словарь с тремя ключами.
2. `print(json.dumps(log, ensure_ascii=False))`.
3. Решение:
```python
import json
payment_id = int(input())
event = input()
log = {"level": "INFO", "event": event, "payment_id": payment_id}
print(json.dumps(log, ensure_ascii=False))
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `9` ⏎ `payment_created` | `{"level": "INFO", "event": "payment_created", "payment_id": 9}` |
| `12` ⏎ `payment_failed` | `{"level": "INFO", "event": "payment_failed", "payment_id": 12}` |

#### ❓ Квиз
Чем структурный (JSON) лог удобнее текстового в проде?
- ✅ `По полям его легко искать и фильтровать в системе мониторинга`
- `Он короче` — не всегда, и дело не в длине.
- `Его не нужно хранить` — хранить логи как раз нужно.
- `Он быстрее пишется` — выигрыш в анализе, а не в скорости записи.

### Урок 15.5 — Health check и метрики
**Экраны:** теория → квиз

#### 📖 Теория
Чтобы знать, что сервис жив, ему делают два «окошка наружу»:
- **health check** — эндпоинт `/health`, который отвечает `{"status": "ok"}`. Системы мониторинга дёргают его каждые несколько секунд: ответил — жив, замолчал — поднимаем тревогу.
- **metrics** — эндпоинт `/metrics` с числами: сколько запросов, сколько ошибок, сколько платежей в обработке. По ним строят графики и **alerts** (оповещения): «ошибок стало слишком много — разбудить дежурного».

🐍 Пай: «/health — пульс сервиса. /metrics — его кардиограмма. Молчит пульс — будят дежурного».

#### ❓ Квиз
Зачем сервису эндпоинт `/health`?
- ✅ `Чтобы мониторинг постоянно проверял, что сервис жив и отвечает`
- `Чтобы показать главную страницу` — это не страница, а проверка статуса.
- `Чтобы хранить логи` — логи хранят отдельно.
- `Чтобы ускорить запросы` — health не про скорость.

### Урок 15.6 — 🏆 БОСС: лог жизни платежа
**Экраны:** теория → задача

#### 📖 Теория
Собери лог одного шага платежа: с уровнем, событием, id платежа и request_id — структурно, в JSON. Именно так выглядит запись, которую потом ищут в мониторинге при разборе инцидента.

🐍 Пай: «Один платёж оставляет цепочку структурных логов. По request_id её собирают воедино — и видят всё, что было».

#### 💻 Задача
**Условие:** собери структурный лог шага платежа. На вход три строки: request_id, событие, id платежа (число). Поля по порядку: `level`=`INFO`, `request_id`, `event`, `payment_id`. Выведи JSON (`ensure_ascii=False`).
**Стартовый код:**
```python
import json
request_id = input()
event = input()
payment_id = int(input())
# собери лог в указанном порядке полей
```
**Подсказки:**
1. Порядок ключей в словаре сохранится в JSON: level, request_id, event, payment_id.
2. `print(json.dumps(log, ensure_ascii=False))`.
3. Решение:
```python
import json
request_id = input()
event = input()
payment_id = int(input())
log = {"level": "INFO", "request_id": request_id, "event": event, "payment_id": payment_id}
print(json.dumps(log, ensure_ascii=False))
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `req-7` ⏎ `sent_to_bank` ⏎ `9` | `{"level": "INFO", "request_id": "req-7", "event": "sent_to_bank", "payment_id": 9}` |
