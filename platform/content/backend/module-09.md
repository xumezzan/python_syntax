**Цель:** ученик умеет вызывать чужие API, ставить timeout, делать retry, обрабатывать ошибки по статус-кодам, проверять ответ и сохранять результат.

### Урок 9.1 — Что такое внешнее API
**Экраны:** теория → квиз

#### 📖 Теория
До этого твой сервер только **отвечал** на запросы. Теперь он сам станет **клиентом** — пойдёт за данными в чужой сервис: курс валют, банк, проверка контрагента.

```python
import httpx
response = httpx.get("https://api.example.com/rate")
data = response.json()
```

Два главных инструмента в Python: `requests` (классика, синхронный) и `httpx` (современный, умеет async). Оба умеют `get`, `post`, читать `.json()` и `.status_code`.

🐍 Пай: «Внешнее API — это когда твой backend сам становится клиентом и стучится к соседям».

#### ❓ Квиз
Твой сервис идёт за курсом валют в чужой сервис. Кто здесь клиент?
- ✅ `Твой backend — он отправляет запрос наружу`
- `Чужой сервис курсов` — он отвечает, значит сервер.
- `Пользователь` — он не участвует в этом запросе между серверами.
- `База данных` — БД тут вообще ни при чём.

### Урок 9.2 — Timeout
**Экраны:** теория → квиз

#### 📖 Теория
Чужой сервис может зависнуть. Если не ограничить ожидание, твой запрос будет ждать вечно — и весь твой сервис встанет.

**Timeout** — максимальное время ожидания ответа:

```python
httpx.get("https://api.example.com/rate", timeout=5)
```

Не дождались за 5 секунд → ошибка таймаута, которую можно поймать и обработать (например, сделать retry).

**Ошибка новичка:** ходить во внешний API без timeout. Один медленный сосед — и твой сервис висит на всех пользователях.

🐍 Пай: «Запрос без timeout — как звонок без сброса. Всегда ставь предел ожидания».

#### ❓ Квиз
Зачем при запросе к внешнему API ставить `timeout`?
- ✅ `Чтобы не ждать зависший сервис вечно и не подвесить свой сервис`
- `Чтобы ускорить чужой сервер` — на скорость чужого сервера ты не влияешь.
- `Чтобы получить больше данных` — timeout про время ожидания, не про объём.
- `Чтобы не платить за запрос` — это не про деньги.

### Урок 9.3 — Обработка ошибок по коду
**Экраны:** теория → задача

#### 📖 Теория
Внешний API отвечает статус-кодом, и реагировать на разные коды нужно по-разному:

- `2xx` — успех, читаем данные;
- `401` — токен протух, надо обновить и повторить;
- `404` — объекта нет, повтор не поможет, пропускаем;
- `5xx` — сервер сбоит, имеет смысл повторить (retry).

Главная мысль: **на ошибки клиента (4xx) повторять бессмысленно** — ты прислал что-то не так, повтор даст ту же ошибку. А вот `5xx` — временная беда сервера, тут retry уместен.

🐍 Пай: «4xx — твоя вина, повтор не спасёт. 5xx — их вина, временная, повтор может помочь».

#### 💻 Задача
**Условие:** по статус-коду ответа выведи действие: `2xx`→`save`, `401`→`reauth`, `404`→`skip`, `5xx`→`retry`. На вход — один код (например `503`).
**Стартовый код:**
```python
code = int(input())
# выведи действие по коду
```
**Подсказки:**
1. Диапазоны удобно ловить через `200 <= code < 300` и `500 <= code < 600`.
2. Отдельные коды 401 и 404 проверь прямым сравнением.
3. Решение:
```python
code = int(input())
if 200 <= code < 300:
    print("save")
elif code == 401:
    print("reauth")
elif code == 404:
    print("skip")
elif 500 <= code < 600:
    print("retry")
else:
    print("skip")
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `200` | `save` |
| `401` | `reauth` |
| `404` | `skip` |
| `503` | `retry` |

### Урок 9.4 — Retry
**Экраны:** теория → задача

#### 📖 Теория
**Retry** — это повтор запроса при временной ошибке. Но повторять надо с умом:
- ограничь число попыток (например, 3) — иначе можно долбить сервис вечно;
- повторяй только на `5xx` и таймаутах;
- на `4xx` не повторяй — бесполезно.

Грамотный retry превращает «сервис на секунду моргнул» в «пользователь ничего не заметил».

🐍 Пай: «Retry — это вторая попытка для невезения. Но не для своих ошибок и не бесконечно».

#### 💻 Задача
**Условие:** тебе дают коды ответов по попыткам (через пробел), например `500 500 200`. Повторяй на `500`; останови и выведи `success N` при первом `200` (N — номер попытки с 1); на `400`/`401`/`404` выведи `client error`; если попытки кончились без успеха — `failed`.
**Стартовый код:**
```python
codes = input().split()
# реализуй логику retry
```
**Подсказки:**
1. Иди по кодам с номером: `for i, code in enumerate(codes, 1):`.
2. На `200` — успех и `break`; на клиентских ошибках — `client error` и `break`; на `500` — продолжай.
3. Решение:
```python
codes = input().split()
result = "failed"
for i, code in enumerate(codes, 1):
    if code == "200":
        result = f"success {i}"
        break
    if code in ("400", "401", "404"):
        result = "client error"
        break
print(result)
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `500 500 200` | `success 3` |
| `200` | `success 1` |
| `401` | `client error` |
| `500 500 500` | `failed` |

### Урок 9.5 — Сохранение и логирование ответа
**Экраны:** теория → задача → квиз

#### 📖 Теория
Получил ответ от внешнего API — не теряй его. В интеграциях принято **сохранять и логировать** каждый обмен: что отправили, что получили, какой код. Это спасает при разборе «почему платёж не прошёл».

Минимальная запись лога интеграции — это словарь:

```python
log = {
    "system": "bank",
    "direction": "outgoing",
    "status_code": 200,
    "response": {"status": "success"},
}
```

Потом такие записи кладут в таблицу `integration_logs` (увидим её в проекте).

🐍 Пай: «Не залогировал обмен — считай, его не было. При разборе инцидента лог — единственный свидетель».

#### 💻 Задача
**Условие:** собери запись лога обмена и выведи её как JSON (`ensure_ascii=False`). Поля: `system` = `bank`, `direction` = `outgoing`, `status_code` = считанное число.
**Стартовый код:**
```python
import json
code = int(input())
# собери словарь лога и выведи как JSON
```
**Подсказки:**
1. Создай словарь с тремя ключами, `status_code` — это `code`.
2. `print(json.dumps(log, ensure_ascii=False))`.
3. Решение:
```python
import json
code = int(input())
log = {"system": "bank", "direction": "outgoing", "status_code": code}
print(json.dumps(log, ensure_ascii=False))
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `200` | `{"system": "bank", "direction": "outgoing", "status_code": 200}` |

#### ❓ Квиз
Зачем сохранять request и response каждого обмена с внешним API?
- ✅ `Чтобы потом разобраться, что и когда отправили и что ответил сервис`
- `Чтобы ускорить запросы` — логи на скорость не влияют.
- `Чтобы не ставить timeout` — это разные вещи, логи timeout не заменяют.
- `Это не нужно` — наоборот, для интеграций логи критичны.

### Урок 9.6 — 🏆 БОСС: вызов платёжного шлюза
**Экраны:** теория → задача

#### 📖 Теория
Собери всё вместе: ты отправляешь платёж в банк, банк может моргнуть `5xx`. Сделай надёжный вызов с retry и реши итог: успех, отказ клиента или провал после всех попыток. Это ядро надёжной интеграции.

🐍 Пай: «Платёж нельзя „почти отправить“. Либо чёткий success, либо честный failed — и всё записано».

#### 💻 Задача
**Условие:** банк отвечает кодами по попыткам (через пробел), максимум 3 попытки. Верни итог платежа: `sent_to_bank` при первом `200`; `failed` при `400` (банк отверг данные); `failed` если все попытки `500`. Учитывай не больше 3 попыток.
**Стартовый код:**
```python
codes = input().split()[:3]
# определи итог платежа
```
**Подсказки:**
1. На `200` → `sent_to_bank` и стоп.
2. На `400` → `failed` и стоп; иначе после цикла → `failed`.
3. Решение:
```python
codes = input().split()[:3]
result = "failed"
for code in codes:
    if code == "200":
        result = "sent_to_bank"
        break
    if code == "400":
        result = "failed"
        break
print(result)
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `500 200` | `sent_to_bank` |
| `400` | `failed` |
| `500 500 500` | `failed` |

### Урок 9.7 — Чеклист модуля
**Экраны:** теория

#### 📖 Теория
**Что ты теперь умеешь:**
- вызывать внешние API как клиент (`requests`/`httpx`);
- ставить timeout и понимать, зачем он;
- реагировать на статус-коды по-разному (save/reauth/skip/retry);
- делать ограниченный retry только на временных ошибках;
- сохранять и логировать каждый обмен.

**Связь с реальной работой:** интеграция с банком или госсистемой — это и есть вызовы внешних API: с токенами, таймаутами, повторами и логами. Когда платёж «завис» — по логам видно, на какой попытке и с каким кодом. Дальше познакомимся с миром, где данные ездят не в JSON, а в XML — SOAP.

🐍 Пай: «С REST-соседями подружились. Теперь — старый, но живой мир корпоратива: SOAP и XML».
