### Урок 17.1 — Что такое ERP
**Экраны:** теория → квиз

#### 📖 Теория
**ERP** (Enterprise Resource Planning) — это большая система управления предприятием. В одной системе крупная компания ведёт финансы, склад, закупки, продажи, персонал, ремонты — всё связано в единое целое.

Аналогия: ERP — это «операционная система предприятия». Как Windows управляет файлами и программами, ERP управляет процессами всей компании.

Зачем это backend-разработчику? Потому что ERP не живёт в вакууме: ему нужно обмениваться данными с банками, налоговой, поставщиками. Этот обмен и строят интеграторы.

🐍 Пай: «ERP — операционная система бизнеса. Всё предприятие в одной системе. А вокруг — мир, с которым её надо связать».

#### ❓ Квиз
Что такое ERP простыми словами?
- ✅ `Единая система управления предприятием: финансы, склад, закупки, персонал`
- `База данных одного отдела` — ERP объединяет всю компанию, а не один отдел.
- `Язык программирования` — это класс систем, а не язык.
- `Тип сервера` — ERP это система, а не железо.

### Урок 17.2 — SAP, S/4HANA и HANA
**Экраны:** теория → квиз

#### 📖 Теория
**SAP** — крупнейший в мире производитель ERP-систем. Когда говорят «у компании SAP», имеют в виду их ERP.

Три слова, которые важно не путать:
- **SAP S/4HANA** — современная ERP-система SAP (текущее поколение). Это то, что внедряют сейчас.
- **SAP HANA** — сверхбыстрая база данных «в памяти», на которой работает S/4HANA. То есть HANA — это БД, а S/4HANA — система поверх неё.
- **SAP-модули** — функциональные части системы (финансы, логистика и т.д.).

🐍 Пай: «HANA — это база (движок). S/4HANA — ERP поверх неё. SAP — компания, которая всё это делает».

#### ❓ Квиз
Чем SAP HANA отличается от SAP S/4HANA?
- ✅ `HANA — это база данных в памяти, S/4HANA — ERP-система, работающая поверх неё`
- `Это одно и то же` — нет, одно база, другое система.
- `HANA — это язык программирования` — HANA это СУБД.
- `S/4HANA старее HANA` — наоборот, S/4HANA это современное поколение поверх HANA.

### Урок 17.3 — Модули SAP
**Экраны:** теория → квиз

#### 📖 Теория
SAP разбит на **функциональные модули** — части под разные процессы:
- **Finance (финансы)** — учёт денег: счета, платежи, бухгалтерия. Здесь рождаются платёжные документы.
- **Logistics (логистика)** — склад, закупки, поставки, движение товаров.
- **Maintenance / ТОиР (обслуживание и ремонт оборудования)** — заявки на ремонт, обслуживание техники, плановые работы.

Для интегратора важнее всего Finance: именно оттуда идут платежи, которые нужно отправлять в банк.

🐍 Пай: «Finance рождает платёж, Logistics двигает товары, Maintenance чинит оборудование. Интегратору чаще всего нужен Finance».

#### ❓ Квиз
Из какого модуля SAP обычно приходят платёжные документы для отправки в банк?
- ✅ `Finance (финансы) — он ведёт деньги и платежи`
- `Logistics` — это про склад и поставки, не про платежи.
- `Maintenance` — это про ремонт оборудования.
- `Ни из какого` — платежи рождаются именно в финансовом модуле.

### Урок 17.4 — Как SAP общается с внешним миром
**Экраны:** теория → пазл

#### 📖 Теория
SAP обменивается данными с другими системами несколькими способами:
- **REST / OData** — современные API SAP (OData — это «диалект» REST от SAP, тоже по сути HTTP + структурированные данные);
- **SOAP** — классические веб-сервисы SAP (XML, как мы изучали);
- **File exchange** — обмен файлами: SAP выкладывает файл (например, реестр платежей), другая система его забирает.

Backend-интегратор должен уметь работать со всеми: где-то REST, где-то SOAP, где-то файлы. Поэтому мы и изучали и JSON, и XML.

🐍 Пай: «SAP говорит на нескольких языках: REST/OData, SOAP, файлы. Хороший интегратор понимает все три».

#### 🧩 Собери код
Расставь путь платёжного документа от SAP до банка по порядку.
**Строки:** `SAP Finance создаёт платёжный документ` · `SAP отдаёт его через API/SOAP/файл` · `Backend принимает и преобразует данные` · `Backend отправляет платёж в банк` · `Банк проводит и отвечает`
**Правильный порядок:** `SAP Finance создаёт платёжный документ` · `SAP отдаёт его через API/SOAP/файл` · `Backend принимает и преобразует данные` · `Backend отправляет платёж в банк` · `Банк проводит и отвечает`

### Урок 17.5 — Роль backend в SAP-интеграциях
**Экраны:** теория → задача → квиз

#### 📖 Теория
Где в этой картине ты? Backend-разработчик — это **слой интеграции** между SAP и внешними системами:

```text
SAP S/4HANA
    ↓ (Finance / Logistics / Maintenance)
API / SOAP / OData / Files
    ↓
Backend Integration Layer  ← ТЫ
    ↓
Bank / Government System / External Service
```

Твоя работа: принять документ из SAP, привести поля к формату банка (data mapping), отправить, обработать ответ, залогировать, защитить от дублей (idempotency), повторить при сбое (retry). Всё, что ты уже изучил, сходится здесь.

**Важнейшее правило:** нельзя бездумно трогать прод. За документами SAP стоят реальные деньги и отчётность. Любое изменение — через тесты, ветку, ревью и аккуратные логи.

🐍 Пай: «Backend — это переводчик и охранник между SAP и банком. Переводит данные и следит, чтобы платёж прошёл ровно один раз».

#### 💻 Задача
**Условие:** SAP Finance прислал платёжный документ: `{"Belnr": "1900000123", "Dmbtr": 500000, "Waers": "UZS"}`. Преобразуй в платёж для банка: `Belnr`→`sap_document_id`, `Dmbtr`→`amount`, `Waers`→`currency`. Выведи JSON (`ensure_ascii=False`).
**Стартовый код:**
```python
import json
doc = json.loads(input())
# преобразуй документ SAP в платёж для банка
```
**Подсказки:**
1. Собери новый словарь с тремя ключами.
2. Значения возьми из полей SAP по их именам.
3. Решение:
```python
import json
doc = json.loads(input())
payment = {
    "sap_document_id": doc["Belnr"],
    "amount": doc["Dmbtr"],
    "currency": doc["Waers"],
}
print(json.dumps(payment, ensure_ascii=False))
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `{"Belnr": "1900000123", "Dmbtr": 500000, "Waers": "UZS"}` | `{"sap_document_id": "1900000123", "amount": 500000, "currency": "UZS"}` |

#### ❓ Квиз
Почему в SAP-интеграциях так важны логи, idempotency и retry?
- ✅ `За документами SAP стоят реальные деньги: платёж должен пройти один раз, быть прослеживаем и переживать сбои`
- `Это требование Python` — нет, это требование к работе с деньгами.
- `Чтобы код был длиннее` — дело не в объёме кода.
- `Это не важно` — наоборот, это критично для финансовых данных.

### Урок 17.6 — 🏆 БОСС: поток SAP → банк
**Экраны:** теория → квиз

🐍 Пай: «Ты видишь всю карту: SAP → backend → банк. Понимаешь и язык данных, и язык денег. Это уровень интегратора».

#### ❓ Квиз
Какова главная роль backend-разработчика в связке SAP ↔ банк?
- ✅ `Надёжно связать системы: преобразовать данные, провести платёж один раз, залогировать и пережить сбои`
- `Настраивать сам SAP изнутри` — это работа функционального консультанта, не backend.
- `Управлять деньгами клиентов` — деньгами управляет банк.
- `Рисовать интерфейс SAP` — интеграция про данные, не про UI.
