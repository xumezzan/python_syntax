### Урок 4.1 — Что такое FastAPI
**Экраны:** теория → квиз

#### 📖 Теория
**FastAPI** — современный Python-фреймворк для создания API. Он популярен, потому что:
- быстрый;
- сам проверяет входные данные;
- сам генерирует документацию (Swagger);
- использует обычные Python-аннотации типов.

Минимальное приложение:

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/health")
def health():
    return {"status": "ok"}
```

Тут `@app.get("/health")` говорит: «когда придёт `GET /health` — вызови эту функцию». Вернёшь словарь — FastAPI сам превратит его в JSON.

🐍 Пай: «FastAPI берёт на себя рутину: маршруты, JSON, проверку данных, документацию. Ты пишешь только логику».

#### ❓ Квиз
Что делает строка `@app.get("/health")` над функцией?
- ✅ `Привязывает функцию к запросу GET /health`
- `Удаляет эндпоинт` — наоборот, она его создаёт.
- `Делает функцию приватной` — декоратор про маршрут, а не про доступ.
- `Запускает сервер` — сервер запускает uvicorn, а не декоратор.

### Урок 4.2 — Запуск и Swagger
**Экраны:** теория → квиз

#### 📖 Теория
FastAPI-приложение запускают сервером **uvicorn**:

```bash
uvicorn app.main:app --reload
```

`app.main:app` значит «в файле `app/main.py` возьми объект `app`». Флаг `--reload` перезапускает сервер при изменении кода — удобно в разработке.

Бонус: открой `http://localhost:8000/docs` — там автоматически появится **Swagger UI**: список всех эндпоинтов, которые можно потыкать прямо в браузере, без Postman.

🐍 Пай: «`/docs` — твоя живая документация. Написал эндпоинт — он сразу там, можно проверить руками».

#### ❓ Квиз
Что появляется по адресу `/docs` у FastAPI-приложения?
- ✅ `Интерактивная документация Swagger со всеми эндпоинтами`
- `Исходный код проекта` — код там не показывается.
- `Логи сервера` — логи идут в консоль, а не на /docs.
- `Главная страница сайта` — /docs это именно документация API.

### Урок 4.3 — Параметры пути и запроса
**Экраны:** теория → баг

#### 📖 Теория
**Path-параметр** объявляют прямо в пути и принимают аргументом функции:

```python
@app.get("/products/{product_id}")
def get_product(product_id: int):
    return {"id": product_id}
```

`{product_id}` в пути → аргумент `product_id: int`. FastAPI сам проверит, что это число.

**Query-параметр** — это просто аргумент функции, которого нет в пути:

```python
@app.get("/products")
def list_products(limit: int = 10):
    return {"limit": limit}
```

Запрос `/products?limit=5` → `limit = 5`. Значение по умолчанию делает параметр необязательным.

🐍 Пай: «Аргумент есть в фигурных скобках пути — это path. Нет — это query».

#### 🐞 Найди баг
**Код:**
```python
@app.get("/products/{product_id}")
def get_product():
    return {"id": product_id}
```
**Ошибка:** строка 2 — path-параметр `{product_id}` объявлен в пути, но не принят в аргументы функции — внутри `product_id` не существует.
**Правильно:** `def get_product(product_id: int):`
**Объяснение после решения:** каждый `{параметр}` из пути нужно объявить аргументом функции — иначе FastAPI не передаст его и переменной не будет.

### Урок 4.4 — Тело запроса и Pydantic
**Экраны:** теория → задача → квиз

#### 📖 Теория
Чтобы принять JSON-тело (например, при создании товара), описывают **Pydantic-схему** — класс с полями и типами:

```python
from pydantic import BaseModel

class ProductIn(BaseModel):
    name: str
    price: float

@app.post("/products")
def create_product(product: ProductIn):
    return {"name": product.name, "price": product.price}
```

Теперь FastAPI сам:
- прочитает JSON из тела;
- проверит, что `name` — строка, а `price` — число;
- если данные плохие — вернёт `422` с понятным описанием, не доходя до твоего кода.

🐍 Пай: «Pydantic-схема — это вахтёр на входе. Кривые данные он не пропустит, и тебе не придётся проверять их руками».

#### 💻 Задача
**Условие (самопроверка):** напиши Pydantic-схему `PaymentIn` с полями: `amount` (целое число), `currency` (строка) и `note` (строка, необязательное, по умолчанию пустая строка).
**Стартовый код:**
```python
from pydantic import BaseModel

class PaymentIn(BaseModel):
    # опиши поля
    pass
```
**Подсказки:**
1. Поле задаётся как `имя: тип`.
2. Значение по умолчанию делает поле необязательным: `note: str = ""`.
3. Решение:
```python
from pydantic import BaseModel

class PaymentIn(BaseModel):
    amount: int
    currency: str
    note: str = ""
```
**Автотесты:** самопроверка: схема описывает три поля, у `note` есть значение по умолчанию.

#### ❓ Квиз
Клиент прислал в теле `{"amount": "много", "currency": "UZS"}`, а в схеме `amount: int`. Что сделает FastAPI?
- ✅ `Вернёт 422 — данные не прошли проверку, до твоего кода дело не дойдёт`
- `Превратит "много" в число` — текст в число не превращается, это ошибка.
- `Вернёт 500` — это была бы ошибка сервера, а тут ошибка клиента (422).
- `Молча подставит 0` — FastAPI не угадывает данные, он их отклоняет.

### Урок 4.5 — Структура проекта
**Экраны:** теория → пазл

#### 📖 Теория
Когда эндпоинтов много, весь код в одном файле превращается в кашу. Поэтому проект делят по слоям — каждый отвечает за своё:

```text
app/
  main.py        ← создаёт приложение, подключает роуты
  config.py      ← настройки (читаются из переменных окружения)
  database.py    ← подключение к БД
  api/routes/    ← эндпоинты (тонкий слой: принять/ответить)
  schemas/       ← Pydantic-схемы (как выглядят данные)
  services/      ← бизнес-логика (что делать)
  models/        ← таблицы БД
  repositories/  ← запросы к БД
tests/           ← тесты
```

Главное правило: **роут тонкий**. Он только принимает запрос и зовёт сервис. Вся настоящая логика — в `services`. Так код легко читать и тестировать.

🐍 Пай: «Роут — официант: принял заказ, передал на кухню. Кухня — это services. Не готовь прямо в зале».

#### 🧩 Собери код
Куда положить каждый кусок? Расставь файлы от «входа запроса» к «данным».
**Строки:** `main.py — приложение` · `api/routes — эндпоинты` · `schemas — проверка данных` · `services — бизнес-логика` · `repositories — запросы к БД`
**Правильный порядок:** `main.py — приложение` · `api/routes — эндпоинты` · `schemas — проверка данных` · `services — бизнес-логика` · `repositories — запросы к БД`

### Урок 4.6 — Настройки через окружение
**Экраны:** теория → квиз

#### 📖 Теория
Никогда не пиши пароли и адреса баз прямо в коде. Их держат в **переменных окружения** и читают через настройки:

```python
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    database_url: str
    secret_key: str

    class Config:
        env_file = ".env"

settings = Settings()
```

В репозитории лежит `.env.example` (с пустыми примерами), а реальный `.env` — только локально и **не коммитится**. Так секреты не утекают в Git.

🐍 Пай: «Секрет в коде = секрет в Git = секрет у всех. Только `.env`, только окружение».

#### ❓ Квиз
Где правильно хранить пароль от базы данных?
- ✅ `В переменных окружения / файле .env, который не попадает в Git`
- `Прямо в config.py` — тогда пароль утечёт в репозиторий.
- `В комментарии рядом с кодом` — это тот же код, секрет утечёт.
- `В README` — README публичный, секретам там не место.

### Урок 4.7 — 🏆 БОСС: каркас Product API
**Экраны:** теория → задача

#### 📖 Теория
Собери минимальный, но настоящий каркас: эндпоинт здоровья и создание товара со схемой. Это фундамент, на котором потом вырастет весь проект.

🐍 Пай: «`/health` есть у каждого живого сервиса — с него начинают и им проверяют, что сервер жив».

#### 💻 Задача
**Условие (самопроверка):** напиши FastAPI-приложение с двумя эндпоинтами: `GET /health` возвращает `{"status": "ok"}`; `POST /products` принимает тело со схемой `ProductIn` (поля `name: str`, `price: float`) и возвращает принятый товар.
**Стартовый код:**
```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

# опиши схему ProductIn и оба эндпоинта
```
**Подсказки:**
1. `@app.get("/health")` → верни словарь со статусом.
2. Схему опиши классом, в `POST` прими её аргументом.
3. Решение:
```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class ProductIn(BaseModel):
    name: str
    price: float

@app.get("/health")
def health():
    return {"status": "ok"}

@app.post("/products", status_code=201)
def create_product(product: ProductIn):
    return {"name": product.name, "price": product.price}
```
**Автотесты:** самопроверка: есть `GET /health` со статусом ok и `POST /products` со схемой и кодом 201.
