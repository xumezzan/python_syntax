### Урок 16.1 — Зачем фоновые задачи
**Экраны:** теория → квиз

#### 📖 Теория
Некоторые операции долгие: отправить платёж в банк, который думает 10 секунд. Если делать это прямо в запросе, пользователь будет ждать ответа все 10 секунд, а сервис — занят.

**Фоновая задача** решает это: запрос быстро отвечает «принято, обрабатываю», а тяжёлую работу делает отдельно, в фоне. Пользователь не ждёт, сервис не блокируется.

```text
без фона:  запрос ──ждём банк 10 сек── ответ   (плохо)
с фоном:   запрос → "принято" сразу; банк обрабатывается в фоне  (хорошо)
```

🐍 Пай: «Долгую работу — в фон. Пользователю — быстрое „принято“. Так сервис остаётся отзывчивым».

#### ❓ Квиз
Зачем выносить отправку платежа в банк в фоновую задачу?
- ✅ `Чтобы запрос быстро ответил, а долгую обработку делать отдельно`
- `Чтобы платёж не сохранялся` — наоборот, он сохраняется, обработка идёт в фоне.
- `Чтобы не логировать` — логи остаются нужны.
- `Чтобы убрать базу данных` — база никуда не девается.

### Урок 16.2 — Что такое очередь
**Экраны:** теория → задача

#### 📖 Теория
**Очередь (queue)** — список задач, которые ждут обработки. Работает по принципу **FIFO** — first in, first out: кто первым встал, того первым и обслужат. Как очередь в магазине.

```text
кладём:  PUT платёж-1, PUT платёж-2
берём:   GET → платёж-1 (он встал первым)
```

Сервис кладёт задачи в очередь, а отдельный «работник» (worker) достаёт их по одной и обрабатывает. Если работа всплеском — очередь сглаживает нагрузку.

🐍 Пай: «Очередь — это честная касса: первым пришёл — первым ушёл. Никто не лезет без очереди».

#### 💻 Задача
**Условие:** реализуй FIFO-очередь. Команды по одной в строке: `PUT x` — положить `x` в конец; `GET` — достать из начала и вывести (или `empty`, если пусто). Читай до конца ввода.
**Стартовый код:**
```python
queue = []
while True:
    try:
        line = input().strip()
    except EOFError:
        break        # команды закончились
    if not line:
        continue
    # обработай PUT и GET
```
**Подсказки:**
1. `PUT x` → `queue.append(x)`.
2. `GET` → если очередь не пуста, `queue.pop(0)` и вывести, иначе `empty`.
3. Решение:
```python
queue = []
while True:
    try:
        line = input().strip()
    except EOFError:
        break
    if not line:
        continue
    if line.startswith("PUT "):
        queue.append(line[4:])
    elif line == "GET":
        print(queue.pop(0) if queue else "empty")
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `PUT a` ⏎ `PUT b` ⏎ `GET` ⏎ `GET` | `a` ⏎ `b` |
| `GET` ⏎ `PUT x` ⏎ `GET` | `empty` ⏎ `x` |

### Урок 16.3 — Redis, Celery, BackgroundTasks
**Экраны:** теория → квиз

#### 📖 Теория
Инструменты для фоновой работы:
- **Redis** — быстрое хранилище в памяти; часто служит самой очередью (туда кладут задачи).
- **Celery** — система фоновых задач: воркеры берут задачи из Redis и выполняют. Подходит для серьёзной нагрузки и расписаний.
- **FastAPI BackgroundTasks** — простой встроенный способ: «ответь клиенту, а потом выполни вот это». Хорош для лёгких задач без отдельной инфраструктуры.

Выбор простой: лёгкая задача — `BackgroundTasks`; серьёзная, с повторами и расписанием — Celery + Redis.

🐍 Пай: «Redis — склад задач. Celery — бригада воркеров. BackgroundTasks — „сделаю по-быстрому сам, после ответа“».

#### ❓ Квиз
Для чего часто используют Redis в фоновой обработке?
- ✅ `Как быструю очередь, куда складывают задачи для воркеров`
- `Как фреймворк для API` — это роль FastAPI, не Redis.
- `Как язык программирования` — Redis это хранилище, не язык.
- `Как систему контроля версий` — это Git, не Redis.

### Урок 16.4 — Retry и failed в фоне
**Экраны:** теория → задача → квиз

#### 📖 Теория
В фоне задача тоже может упасть (банк недоступен). Тогда её **повторяют**, но с ограничением: например, до 3 попыток. Если все попытки провалились — задачу помечают `failed` и откладывают на ручной разбор (помнишь dead letter queue?).

```text
попытка 1: fail → retry
попытка 2: fail → retry
попытка 3: fail → статус failed (хватит)
```

Так задача либо в итоге выполнится, либо честно станет `failed` — но не зависнет навсегда и не будет долбить банк бесконечно.

🐍 Пай: «3 попытки — и хватит. Получилось — done. Нет — failed и на полку. Никаких вечных повторов».

#### 💻 Задача
**Условие:** фоновая задача отправляет платёж. На вход — исходы попыток через пробел (`ok`/`fail`), максимум 3 попытки. Верни `done` при первом `ok`; если за 3 попытки успеха нет — `failed`.
**Стартовый код:**
```python
attempts = input().split()[:3]
# определи итог фоновой задачи
```
**Подсказки:**
1. Иди по попыткам: на `ok` → `done` и стоп.
2. Если цикл закончился без `ok` → `failed`.
3. Решение:
```python
attempts = input().split()[:3]
result = "failed"
for outcome in attempts:
    if outcome == "ok":
        result = "done"
        break
print(result)
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `fail ok` | `done` |
| `fail fail fail` | `failed` |
| `ok` | `done` |

#### ❓ Квиз
Фоновая задача упала 3 раза подряд при лимите 3 попытки. Что с ней делать?
- ✅ `Пометить failed и отложить на ручной разбор, не повторять бесконечно`
- `Повторять, пока не получится` — это перегрузит систему.
- `Удалить и забыть` — потеря задачи недопустима.
- `Считать выполненной` — она не выполнилась, это была бы ложь.

### Урок 16.5 — 🏆 БОСС: очередь платежей с retry
**Экраны:** теория → задача

#### 📖 Теория
Собери ядро фоновой обработки платежей: задача с попытками и лимитом. Веди счётчик повторов и реши итог. Это то, что в реальном сервисе делает воркер для каждого платежа из очереди.

🐍 Пай: «Воркер берёт платёж, пробует, считает попытки. done или failed — но всегда определённость и всегда в логах».

#### 💻 Задача
**Условие:** воркер обрабатывает платёж. На вход — исходы попыток через пробел (`ok`/`fail`), лимит 3. Выведи две строки: число использованных попыток и итог (`done` при первом `ok`, иначе `failed`). На `ok` останавливайся сразу.
**Стартовый код:**
```python
attempts = input().split()
used = 0
result = "failed"
# посчитай попытки и определи итог (лимит 3)
```
**Подсказки:**
1. Иди по попыткам, увеличивай `used`, но не больше 3.
2. На `ok` → `done` и стоп.
3. Решение:
```python
attempts = input().split()
used = 0
result = "failed"
for outcome in attempts[:3]:
    used += 1
    if outcome == "ok":
        result = "done"
        break
print(used)
print(result)
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `fail ok` | `2` ⏎ `done` |
| `fail fail fail` | `3` ⏎ `failed` |
| `ok` | `1` ⏎ `done` |
