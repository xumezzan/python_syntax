### Урок 3.1 — Что такое API
**Экраны:** теория → квиз

#### 📖 Теория
**API** — это «розетка» программы. Тебе не нужно знать, как внутри устроен сервис, — есть понятные точки подключения, через которые можно получить данные или попросить что-то сделать.

Аналогия: пульт от телевизора. Кнопки — это API. Ты нажимаешь «громче», не вскрывая телевизор и не зная, как там работают микросхемы.

Когда приложение такси показывает машины на карте — оно дёргает API сервера такси. Когда сайт показывает погоду — он спрашивает погодное API.

🐍 Пай: «API — это договор: „вот что у меня можно попросить и что я верну в ответ“».

#### ❓ Квиз
Приложение погоды показывает прогноз. Что такое API в этой истории?
- ✅ `Набор запросов к серверу погоды, через которые приложение получает прогноз`
- `Сама база данных с погодой` — данные хранит БД, а API — способ их попросить.
- `Дизайн приложения` — внешний вид к API отношения не имеет.
- `Интернет-соединение` — соединение лишь канал, а API — правила запросов.

### Урок 3.2 — Что такое REST
**Экраны:** теория → квиз

#### 📖 Теория
**REST** — это популярный стиль построения API. Главная идея: всё вокруг **ресурсов** (товары, заказы, платежи), и над каждым ресурсом выполняют стандартные действия HTTP-методами.

- **Ресурс** — сущность: `товар`, `заказ`, `пользователь`.
- **Endpoint** — конкретный адрес для работы с ресурсом: `/products`, `/products/42`.

Принцип REST: один и тот же путь `/products`, но разные методы делают разное:

```text
GET    /products       → получить список
POST   /products       → создать товар
GET    /products/42    → получить товар №42
PATCH  /products/42    → изменить товар №42
DELETE /products/42    → удалить товар №42
```

🐍 Пай: «REST = ресурсы + методы. Путь говорит „над чем“, метод — „что сделать“».

#### ❓ Квиз
Что такое «ресурс» в REST?
- ✅ `Сущность, которой управляет API: товар, заказ, платёж`
- `HTTP-метод` — метод это действие, а ресурс — объект действия.
- `Статус-код ответа` — код это итог, а не ресурс.
- `Имя сервера` — сервер хостит API, но ресурс это сущность внутри.

### Урок 3.3 — CRUD
**Экраны:** теория → задача

#### 📖 Теория
**CRUD** — четыре базовые операции над данными. Их хватает для большинства API:

| Буква | Действие | HTTP-метод |
|-------|----------|-----------|
| C — Create | создать | `POST` |
| R — Read | прочитать | `GET` |
| U — Update | изменить | `PUT` / `PATCH` |
| D — Delete | удалить | `DELETE` |

Если ты умеешь делать CRUD над ресурсом — ты умеешь делать REST API.

🐍 Пай: «CRUD — это алфавит backend. Создать, прочитать, обновить, удалить — и так с любой сущностью».

#### 💻 Задача
**Условие:** по букве CRUD-операции выведи HTTP-метод. Вход: `C`, `R`, `U` или `D`. Выход: `POST`, `GET`, `PATCH`, `DELETE`.
**Стартовый код:**
```python
op = input()
# выведи метод для операции CRUD
```
**Подсказки:**
1. Заведи словарь `{"C": "POST", "R": "GET", "U": "PATCH", "D": "DELETE"}`.
2. Достань значение по ключу.
3. Решение:
```python
op = input()
methods = {"C": "POST", "R": "GET", "U": "PATCH", "D": "DELETE"}
print(methods[op])
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `C` | `POST` |
| `R` | `GET` |
| `D` | `DELETE` |

### Урок 3.4 — Правильные статус-коды
**Экраны:** теория → задача → квиз

#### 📖 Теория
Хороший REST API возвращает не просто `200` на всё подряд, а точный код:

- создан объект → `201 Created`;
- успешно получили/обновили → `200 OK`;
- удалили, отвечать нечем → `204 No Content`;
- объект не найден → `404 Not Found`;
- данные не прошли проверку → `422 Unprocessable Entity`.

**Ошибка новичка:** отвечать `200` даже когда товар не найден, и класть ошибку в тело. Клиенту тогда трудно понять, успех это или нет. Код должен честно говорить итог.

🐍 Пай: «Статус-код — это честность API. Создал — 201, не нашёл — 404. Не ври клиенту двумястами на всё».

#### 💻 Задача
**Условие:** по названию ситуации выведи статус-код. Вход — одно из: `created`, `ok`, `not_found`, `invalid`. Выход — `201`, `200`, `404`, `422`.
**Стартовый код:**
```python
case = input()
# выведи подходящий статус-код
```
**Подсказки:**
1. Словарь ситуация → код.
2. `codes[case]`.
3. Решение:
```python
case = input()
codes = {"created": "201", "ok": "200", "not_found": "404", "invalid": "422"}
print(codes[case])
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `created` | `201` |
| `not_found` | `404` |
| `invalid` | `422` |

#### ❓ Квиз
Клиент сделал `POST /products` и товар успешно создан. Какой статус-код правильный?
- ✅ `201 Created`
- `200 OK` — формально не ошибка, но для созданного объекта точнее `201`.
- `204 No Content` — это для случая «успех, но тела нет», например после удаления.
- `404 Not Found` — это ошибка, а у нас успех.

### Урок 3.5 — Эндпоинты Product API
**Экраны:** теория → пазл → баг

#### 📖 Теория
Спроектируем настоящий маленький API для товаров. Полный набор эндпоинтов:

```text
GET    /products       список товаров
POST   /products       создать товар
GET    /products/{id}  один товар
PATCH  /products/{id}  изменить товар
DELETE /products/{id}  удалить товар
```

Обрати внимание: список и создание — на `/products` (без id), а операции с конкретным товаром — на `/products/{id}` (с id в пути). Это и есть «ресурсный» стиль REST.

🐍 Пай: «Без id — про коллекцию. С id — про один элемент. Простое правило, работает всегда».

#### 🧩 Собери код
Расставь эндпоинты Product API в логичном порядке: сначала операции над коллекцией, потом над одним товаром.
**Строки:** `GET /products` · `POST /products` · `GET /products/{id}` · `PATCH /products/{id}` · `DELETE /products/{id}`
**Правильный порядок:** `GET /products` · `POST /products` · `GET /products/{id}` · `PATCH /products/{id}` · `DELETE /products/{id}`

#### 🐞 Найди баг
**Код:**
```python
# хотим получить ОДИН товар по id
# дизайнер написал эндпоинт так:
endpoint = "GET /products"
# но это вернёт весь список, а не товар №42
```
**Ошибка:** строка 3 — для одного товара нужен id в пути: `/products/42`, а не общий `/products`.
**Правильно:** `endpoint = "GET /products/42"`
**Объяснение после решения:** `/products` — это коллекция (весь список). Один элемент достаётся по пути с id: `/products/{id}`.

### Урок 3.6 — 🏆 БОСС: симуляция Product API
**Экраны:** теория → задача

#### 📖 Теория
Сервер хранит товары в памяти — в словаре `{id: название}`. Тебе приходят команды, по одной в строке:
- `ADD название` — создать товар (id выдаётся по порядку: 1, 2, 3...), вывести `created <id>`;
- `GET id` — вывести название товара или `404`, если такого нет.

Это упрощённая модель того, что делает настоящий REST API: принимает запросы и меняет/читает хранилище.

🐍 Пай: «Внутри любого CRUD-API — вот такое хранилище и обработчик команд. Ты только что собрал его суть».

#### 💻 Задача
**Условие:** обработай команды (по одной в строке). На `ADD имя` создай товар и выведи `created N` (N — новый id, начиная с 1). На `GET N` выведи имя товара или `404`. Команды читай до конца ввода.
**Стартовый код:**
```python
products = {}
next_id = 1
while True:
    try:
        line = input().strip()
    except EOFError:
        break        # ввод закончился
    if not line:
        continue
    # разбери команду и ответь
```
**Подсказки:**
1. Раздели строку: `cmd, arg = line.split(" ", 1)`.
2. Для `ADD`: положи `products[next_id] = arg`, выведи `created`, увеличь `next_id`.
3. Решение:
```python
products = {}
next_id = 1
while True:
    try:
        line = input().strip()
    except EOFError:
        break
    if not line:
        continue
    cmd, arg = line.split(" ", 1)
    if cmd == "ADD":
        products[next_id] = arg
        print("created", next_id)
        next_id += 1
    elif cmd == "GET":
        pid = int(arg)
        print(products.get(pid, "404"))
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `ADD Хлеб` ⏎ `GET 1` | `created 1` ⏎ `Хлеб` |
| `ADD Сок` ⏎ `GET 5` | `created 1` ⏎ `404` |
