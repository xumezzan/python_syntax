### Урок 12.1 — Зачем тесты
**Экраны:** теория → квиз

#### 📖 Теория
**Тест** — это код, который проверяет твой код. Запустил тесты — за секунду узнал, не сломалось ли что-то после правок.

Два главных вида:
- **Unit-тест** — проверяет одну маленькую функцию в изоляции (например, маппинг полей).
- **Integration-тест** — проверяет связку: эндпоинт + база + логика вместе.

Аналогия: unit-тест — проверить одну деталь двигателя. Integration-тест — завести машину и проехать.

🐍 Пай: «Без тестов каждая правка — лотерея. С тестами — нажал «прогнать» и спокоен».

#### ❓ Квиз
Чем unit-тест отличается от integration-теста?
- ✅ `Unit проверяет одну функцию в изоляции, integration — связку компонентов вместе`
- `Ничем, это синонимы` — нет, у них разный масштаб проверки.
- `Unit медленнее` — наоборот, unit обычно быстрее.
- `Integration не нужен` — нужен: связки ломаются чаще одиночных функций.

### Урок 12.2 — pytest и assert
**Экраны:** теория → задача

#### 📖 Теория
В Python тесты пишут на **pytest**. Тест — это функция с именем на `test_` и проверкой `assert`:

```python
def test_add():
    assert add(2, 3) == 5
```

`assert` — «утверждаю, что это правда». Если правда — тест молча проходит. Если нет — тест падает и показывает, что пошло не так.

Запуск: команда `pytest` находит все `test_*` и прогоняет их.

🐍 Пай: «assert — это проверка обещания. Код обещал вернуть 5? Assert ловит его на слове».

#### 💻 Задача
**Условие:** тест уже написан — реализуй функцию `is_success(status)`, чтобы он прошёл. Функция возвращает `True`, если статус равен `"success"`, иначе `False`. Если тест пройдёт, программа выведет `OK`.
**Стартовый код:**
```python
def is_success(status):
    # верни True, если статус "success"
    pass

# готовый тест — не меняй:
assert is_success("success") is True
assert is_success("failed") is False
print("OK")
```
**Подсказки:**
1. Достаточно вернуть результат сравнения: `status == "success"`.
2. Сравнение само даёт `True`/`False`.
3. Решение:
```python
def is_success(status):
    return status == "success"
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| — | `OK` |

### Урок 12.3 — FastAPI TestClient
**Экраны:** теория → квиз

#### 📖 Теория
Чтобы протестировать эндпоинты, не поднимая сервер, у FastAPI есть **TestClient**. Он шлёт запросы прямо в приложение в памяти:

```python
from fastapi.testclient import TestClient
client = TestClient(app)

def test_health():
    response = client.get("/health")
    assert response.status_code == 200
    assert response.json() == {"status": "ok"}
```

Проверяют обычно две вещи: **status_code** (правильный ли код) и **json()** (правильное ли тело).

🐍 Пай: «TestClient — это игрушечный клиент. Он стучится в твоё приложение без настоящей сети и сервера».

#### ❓ Квиз
Что обычно проверяют в тесте эндпоинта через TestClient?
- ✅ `Статус-код ответа и содержимое JSON-тела`
- `Скорость интернета` — TestClient вообще не ходит в сеть.
- `Дизайн страницы` — у API нет страниц, есть данные.
- `Версию Python` — это не предмет теста эндпоинта.

### Урок 12.4 — Моки внешних API
**Экраны:** теория → задача → квиз

#### 📖 Теория
В тестах нельзя дёргать настоящий банк — это деньги и нестабильность. Вместо этого внешний вызов **мокают** — подменяют заглушкой, которая возвращает заранее заданный ответ.

```text
тест → вместо реального банка → mock, который вернёт {"status": "success"}
```

Так тест становится быстрым, стабильным и безопасным: ты проверяешь свою логику, а не работу банка.

🐍 Пай: «Mock — это дублёр банка на съёмках теста. Выглядит как банк, отвечает как банк, но денег не трогает».

#### 💻 Задача
**Условие:** напиши `fake_bank(payment)` — мок банка, который всегда возвращает словарь `{"status": "success", "amount": payment["amount"]}`. Тест проверит его. При успехе выведется `OK`.
**Стартовый код:**
```python
def fake_bank(payment):
    # верни фиктивный успешный ответ банка
    pass

# готовый тест — не меняй:
result = fake_bank({"amount": 500})
assert result["status"] == "success"
assert result["amount"] == 500
print("OK")
```
**Подсказки:**
1. Верни словарь с двумя ключами.
2. `amount` возьми из `payment["amount"]`.
3. Решение:
```python
def fake_bank(payment):
    return {"status": "success", "amount": payment["amount"]}
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| — | `OK` |

#### ❓ Квиз
Зачем мокать банк в тестах вместо реального вызова?
- ✅ `Чтобы тесты были быстрыми, стабильными и не трогали настоящие деньги`
- `Чтобы тесты шли медленнее` — наоборот, мок их ускоряет.
- `Потому что банк нельзя протестировать` — можно, но не в каждом прогоне тестов.
- `Чтобы не писать assert` — assert всё равно нужен.

### Урок 12.5 — Фикстуры и тестовая БД
**Экраны:** теория → квиз

#### 📖 Теория
Тестам часто нужны одинаковые заготовки: подключение к БД, тестовый пользователь. В pytest их создают через **фикстуры** — функции с `@pytest.fixture`, результат которых тест получает аргументом.

```python
@pytest.fixture
def client():
    return TestClient(app)

def test_health(client):     # client придёт из фикстуры
    assert client.get("/health").status_code == 200
```

И ещё важное правило: тесты ходят в **отдельную тестовую БД**, а не в боевую. После теста данные откатывают, чтобы тесты не мешали друг другу и не портили прод.

🐍 Пай: «Фикстура — готовый набор для теста. Тестовая БД — песочница: наигрался и стёр, прод не тронут».

#### ❓ Квиз
Почему тесты должны работать с отдельной тестовой базой, а не с боевой?
- ✅ `Чтобы не испортить и не засорить реальные данные пользователей`
- `Тестовая база быстрее` — дело не в скорости, а в безопасности данных.
- `Боевую базу нельзя читать` — можно, но писать в неё тестами опасно.
- `Так требует Python` — это не требование языка, а здравый смысл.

### Урок 12.6 — 🏆 БОСС: тест идемпотентности
**Экраны:** теория → задача

#### 📖 Теория
Самое ценное — протестировать защиту от дублей. Реализуй обработчик платежей под готовый тест: первый раз с ключом — `created`, повтор — `duplicate`. Если логика верна, тест пройдёт.

🐍 Пай: «Тест на идемпотентность — главный страж денег. Прошёл — значит, дубль платежа невозможен».

#### 💻 Задача
**Условие:** реализуй класс `PaymentService` с методом `process(key)`: на новый ключ верни `"created"`, на уже виденный — `"duplicate"`. Готовый тест проверит поведение и выведет `OK`.
**Стартовый код:**
```python
class PaymentService:
    def __init__(self):
        self.seen = set()

    def process(self, key):
        # верни "created" или "duplicate"
        pass

# готовый тест — не меняй:
s = PaymentService()
assert s.process("ABC") == "created"
assert s.process("ABC") == "duplicate"
assert s.process("XYZ") == "created"
print("OK")
```
**Подсказки:**
1. Проверь `if key in self.seen`.
2. Новый ключ добавь в `self.seen` и верни `"created"`.
3. Решение:
```python
def process(self, key):
    if key in self.seen:
        return "duplicate"
    self.seen.add(key)
    return "created"
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| — | `OK` |
