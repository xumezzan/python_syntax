### Урок 11.1 — Что такое интеграция
**Экраны:** теория → квиз

#### 📖 Теория
**Интеграция** — это когда одна система передаёт данные другой и они работают вместе. Например: SAP создаёт платёж → твой backend → банк проводит деньги → ответ возвращается обратно в SAP.

```text
Система A  →  Backend (ты)  →  Система B
   SAP          мост            Банк
```

Backend-интегратор — это **мост** между системами. Он принимает данные у одной, приводит их к формату другой, отправляет, получает ответ и сообщает результат обратно.

🐍 Пай: «Интеграция — это перевод между двумя системами, которые говорят на разных языках. Ты — переводчик».

#### ❓ Квиз
Что делает backend-интегратор в связке SAP → банк?
- ✅ `Принимает данные у одной системы, приводит к формату другой и передаёт дальше`
- `Заменяет банк` — нет, банк проводит деньги, backend только посредник.
- `Хранит деньги пользователей` — деньги в банке, а не в backend.
- `Рисует интерфейс SAP` — интеграция про данные, а не про UI.

### Урок 11.2 — Data mapping
**Экраны:** теория → задача

#### 📖 Теория
Системы называют одни и те же данные по-разному. SAP шлёт `Waers`, а банк ждёт `currency`. **Data mapping** — перекладывание полей из формата одной системы в формат другой.

```text
SAP                 Банк
Dmbtr   →   amount
Waers   →   currency
```

Маппинг — это словарь соответствий. Без него системы друг друга не поймут.

**Ошибка новичка:** передать чужие имена полей как есть. Банк не знает, что такое `Waers`, — он ждёт `currency`.

🐍 Пай: «Маппинг — это таможня данных. На входе одни имена полей, на выходе — те, что ждёт принимающая система».

#### 💻 Задача
**Условие:** SAP прислал платёж в своём формате: `{"Dmbtr": 500000, "Waers": "UZS"}`. Преобразуй в формат банка: `Dmbtr`→`amount`, `Waers`→`currency`. Выведи результат как JSON (`ensure_ascii=False`).
**Стартовый код:**
```python
import json
sap = json.loads(input())
# собери словарь в формате банка и выведи как JSON
```
**Подсказки:**
1. Создай новый словарь с ключами `amount` и `currency`.
2. Значения возьми из `sap["Dmbtr"]` и `sap["Waers"]`.
3. Решение:
```python
import json
sap = json.loads(input())
bank = {"amount": sap["Dmbtr"], "currency": sap["Waers"]}
print(json.dumps(bank, ensure_ascii=False))
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `{"Dmbtr": 500000, "Waers": "UZS"}` | `{"amount": 500000, "currency": "UZS"}` |
| `{"Dmbtr": 100, "Waers": "USD"}` | `{"amount": 100, "currency": "USD"}` |

### Урок 11.3 — Идемпотентность
**Экраны:** теория → задача → квиз

#### 📖 Теория
Сеть ненадёжна. Запрос на платёж может случайно уйти дважды (повтор, retry, двойной клик). Если не защититься — спишутся деньги дважды. Это недопустимо.

**Идемпотентность** — гарантия, что повтор одной и той же операции не создаст дубль. Достигается через **idempotency_key**: клиент шлёт уникальный ключ операции, сервер запоминает уже обработанные ключи. Пришёл повтор с тем же ключом — сервер не создаёт новый платёж, а возвращает прежний результат.

```text
ключ ABC, первый раз  →  создаём платёж
ключ ABC, второй раз  →  уже видели, дубль не создаём
```

🐍 Пай: «Idempotency_key — это паспорт операции. Один паспорт — один платёж, сколько бы раз он ни пришёл».

#### 💻 Задача
**Условие:** тебе приходят idempotency-ключи платежей, по одному в строке. На новый ключ выведи `created`, на уже виденный — `duplicate`. Читай до конца ввода.
**Стартовый код:**
```python
seen = set()
while True:
    try:
        key = input().strip()
    except EOFError:
        break        # ключи закончились
    if not key:
        continue
    # выведи created или duplicate
```
**Подсказки:**
1. Множество `seen` хранит уже обработанные ключи.
2. Если `key in seen` — это `duplicate`, иначе `created` и добавь ключ в `seen`.
3. Решение:
```python
seen = set()
while True:
    try:
        key = input().strip()
    except EOFError:
        break
    if not key:
        continue
    if key in seen:
        print("duplicate")
    else:
        seen.add(key)
        print("created")
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `ABC` ⏎ `ABC` | `created` ⏎ `duplicate` |
| `A` ⏎ `B` ⏎ `A` | `created` ⏎ `created` ⏎ `duplicate` |

#### ❓ Квиз
Зачем платежу нужен idempotency_key?
- ✅ `Чтобы повторная отправка того же платежа не создала дубль и не списала деньги дважды`
- `Чтобы ускорить платёж` — ключ про защиту от дублей, не про скорость.
- `Чтобы зашифровать сумму` — это не про шифрование.
- `Чтобы узнать баланс` — ключ не имеет отношения к балансу.

### Урок 11.4 — Webhook
**Экраны:** теория → квиз

#### 📖 Теория
Иногда ответ приходит не сразу. Банк принял платёж в обработку и ответит позже. Два способа узнать итог:
- **polling** — самому дёргать банк «ну что, готово?» каждые N секунд (неэффективно);
- **webhook** — банк сам постучится к тебе, когда будет готово.

**Webhook** — это твой эндпоинт, который вызывает чужая система, когда у неё есть новость:

```text
POST /bank/webhook
{"payment_id": 9, "status": "success"}
```

Получив webhook, ты обновляешь статус платежа у себя. Это «не звони мне — я сам позвоню».

🐍 Пай: «Polling — стучаться в дверь каждую минуту. Webhook — оставить телефон, и тебе перезвонят. Второе — взрослее».

#### ❓ Квиз
Что такое webhook?
- ✅ `Твой эндпоинт, который внешняя система вызывает сама, когда есть новость`
- `Запрос, который ты шлёшь каждую секунду` — это polling, противоположность.
- `Тип базы данных` — webhook не про хранение.
- `Формат XML` — это не формат, а способ уведомления.

### Урок 11.5 — Audit trail и integration log
**Экраны:** теория → задача

#### 📖 Теория
В деньгах важно мочь ответить на вопрос «что именно произошло и когда». Для этого ведут **audit trail** — журнал всех обменов. Каждая запись `integration_log` фиксирует:

```text
system_name      — с кем общались (bank, sap)
direction        — incoming / outgoing
request_payload  — что отправили
response_payload — что получили
status_code      — код ответа
error_message    — текст ошибки, если была
created_at       — когда
```

Если платёж потерялся — поднимаешь логи и видишь весь путь. Без audit trail разбор инцидента превращается в гадание.

🐍 Пай: «Audit trail — чёрный ящик интеграции. Случилось ЧП — открыл, и видно всё по шагам».

#### 💻 Задача
**Условие:** собери запись integration_log для исходящего вызова банка. На вход — две строки: статус-код и текст ошибки (или `-`, если ошибки нет). Поля: `system_name`=`bank`, `direction`=`outgoing`, `status_code`=число, `error_message`=текст или `null`, если `-`. Выведи JSON (`ensure_ascii=False`).
**Стартовый код:**
```python
import json
code = int(input())
error = input()
# error == "-" значит ошибки нет (None)
```
**Подсказки:**
1. Если `error == "-"`, замени его на `None`.
2. `None` в JSON станет `null` автоматически.
3. Решение:
```python
import json
code = int(input())
error = input()
log = {
    "system_name": "bank",
    "direction": "outgoing",
    "status_code": code,
    "error_message": None if error == "-" else error,
}
print(json.dumps(log, ensure_ascii=False))
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `200` ⏎ `-` | `{"system_name": "bank", "direction": "outgoing", "status_code": 200, "error_message": null}` |
| `500` ⏎ `timeout` | `{"system_name": "bank", "direction": "outgoing", "status_code": 500, "error_message": "timeout"}` |

### Урок 11.6 — Retry, dead letter, сверка
**Экраны:** теория → квиз

#### 📖 Теория
Три понятия, которые делают интеграцию надёжной:
- **Retry** — повтор при временной ошибке (мы это уже умеем).
- **Dead letter queue** — «полка для безнадёжных». Если после всех попыток операция так и не прошла, её не теряют, а откладывают в отдельную очередь — разберётся человек.
- **Reconciliation (сверка)** — периодическая проверка «совпадает ли картина у нас и у банка». Например, в конце дня сверяют список проведённых платежей по обеим сторонам — нет ли расхождений.

🐍 Пай: «Retry — вторая попытка. Dead letter — полка для безнадёжных. Сверка — вечерняя ревизия, всё ли сошлось».

#### ❓ Квиз
Платёж не прошёл даже после всех retry. Куда его деть, чтобы не потерять?
- ✅ `В dead letter queue — отложить для ручного разбора`
- `Удалить` — потеря платежа недопустима.
- `Повторять бесконечно` — это перегрузит систему и не поможет.
- `Игнорировать` — нерешённый платёж нельзя просто забыть.

### Урок 11.7 — 🏆 БОСС: платёж без дублей
**Экраны:** теория → задача

#### 📖 Теория
Собери ядро интеграции: поток платёжных запросов с ключами идемпотентности. Новый ключ → создаём платёж и присваиваем id по порядку. Повтор → возвращаем тот же id, не создавая дубль. Это ровно то, что защищает реальные деньги.

🐍 Пай: «Один ключ — один платёж — один id. Сколько бы раз ни пришёл повтор, дубля не будет. Это и есть надёжность».

#### 💻 Задача
**Условие:** приходят idempotency-ключи (по одному в строке). На новый ключ выдай новый id (с 1) и выведи `created <id>`. На повтор выведи `duplicate <id>` с тем же id, что выдали в первый раз. Читай до конца ввода.
**Стартовый код:**
```python
ids = {}
next_id = 1
while True:
    try:
        key = input().strip()
    except EOFError:
        break        # ключи закончились
    if not key:
        continue
    # выдай новый id или верни существующий
```
**Подсказки:**
1. Словарь `ids` хранит `ключ → id`.
2. Если ключ уже есть — `duplicate` с `ids[key]`; иначе присвой `next_id`, сохрани, увеличь.
3. Решение:
```python
ids = {}
next_id = 1
while True:
    try:
        key = input().strip()
    except EOFError:
        break
    if not key:
        continue
    if key in ids:
        print("duplicate", ids[key])
    else:
        ids[key] = next_id
        print("created", next_id)
        next_id += 1
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `ABC` ⏎ `ABC` | `created 1` ⏎ `duplicate 1` |
| `A` ⏎ `B` ⏎ `A` | `created 1` ⏎ `created 2` ⏎ `duplicate 1` |
