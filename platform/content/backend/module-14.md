### Урок 14.1 — Что такое Git
**Экраны:** теория → квиз

#### 📖 Теория
**Git** — система контроля версий. Он помнит историю всех изменений кода: кто, что и когда поменял. Можно вернуться к любой прошлой версии, сравнить, понять, что сломалось.

Главные единицы:
- **commit** — сохранённый снимок изменений с подписью «что я сделал»;
- **branch (ветка)** — отдельная линия разработки, чтобы не мешать чужой работе;
- **merge** — слияние ветки обратно в основную.

Аналогия: Git — это машина времени для кода. Каждый commit — точка сохранения, в которую можно вернуться.

🐍 Пай: «Git помнит всё. Сломал — откатился. Поэтому смело экспериментируй в своей ветке».

#### ❓ Квиз
Что такое commit в Git?
- ✅ `Сохранённый снимок изменений с описанием, что было сделано`
- `Удаление кода` — commit, наоборот, сохраняет состояние.
- `Запуск программы` — Git не запускает код.
- `Ветка разработки` — ветка это branch, а commit — точка в истории.

### Урок 14.2 — Ветки и pull request
**Экраны:** теория → пазл

#### 📖 Теория
В команде не пишут прямо в основную ветку (`main`). Делают так:
1. создают **ветку** под задачу: `git checkout -b feature/payments`;
2. коммитят изменения в неё;
3. пушат ветку и открывают **pull request (PR)** — предложение влить изменения в `main`;
4. коллеги делают **code review** — читают и комментируют;
5. после одобрения PR **мёржат** в `main`.

Так `main` всегда остаётся рабочим, а новые фичи проходят проверку перед вливанием.

🐍 Пай: «Своя ветка → PR → ревью → merge. Эта цепочка бережёт main от случайных поломок».

### Урок 14.3 — Хороший commit message
**Экраны:** теория → задача → квиз

#### 📖 Теория
Commit message — это записка будущим себе и команде. Плохой: `fix`, `updates`, `asdf`. Хороший — коротко и по делу, что изменилось:

```text
add idempotency key to payments
fix 500 error on empty bank response
update README with docker instructions
```

Правила: кратко, в настоящем времени, по сути изменения. Через полгода ты скажешь себе спасибо.

🐍 Пай: «Через месяц `fix` не скажет тебе ничего. `fix 500 on empty bank response` — скажет всё».

#### 💻 Задача
**Условие:** выбери хороший commit message. На вход — действие через двоеточие: тип и суть, например `add:idempotency key`. Выведи их в формате `<тип>: <суть>` (с пробелом после двоеточия).
**Стартовый код:**
```python
line = input()  # например: add:idempotency key
# выведи в формате "тип: суть"
```
**Подсказки:**
1. Раздели строку по первому двоеточию: `line.split(":", 1)`.
2. Собери обратно с пробелом: `f"{kind}: {text}"`.
3. Решение:
```python
line = input()
kind, text = line.split(":", 1)
print(f"{kind.strip()}: {text.strip()}")
```
**Автотесты:**
| Ввод | Ожидаемый вывод |
|------|------------------|
| `add:idempotency key` | `add: idempotency key` |
| `fix:500 on empty response` | `fix: 500 on empty response` |

#### ❓ Квиз
Какой commit message хороший?
- ✅ `fix 500 error on empty bank response`
- `fix` — слишком общо, непонятно, что именно поправили.
- `asdfgh` — бессмысленный набор символов.
- `изменения` — не говорит, какие именно изменения.

### Урок 14.4 — .gitignore и .env.example
**Экраны:** теория → квиз

#### 📖 Теория
В Git нельзя класть что попало. Два важных файла:
- **.gitignore** — список того, что Git должен игнорировать: `.env`, `__pycache__/`, `*.pyc`, виртуальное окружение. Секреты и мусор в репозиторий не попадают.
- **.env.example** — шаблон переменных окружения с пустыми/примерными значениями. Его коммитят, чтобы коллега знал, какие переменные нужны, но без реальных секретов.

```text
.env           → в .gitignore, НЕ коммитится (реальные секреты)
.env.example   → коммитится (шаблон без секретов)
```

**Ошибка новичка:** закоммитить `.env` с реальными паролями. Они навсегда останутся в истории Git, даже если потом удалить файл.

🐍 Пай: «.env — в .gitignore. .env.example — в репозиторий. Шаблон делись, секреты держи при себе».

#### ❓ Квиз
Какой файл коммитят в репозиторий, а какой — нет?
- ✅ `.env.example коммитят (шаблон), .env — нет (реальные секреты)`
- `Оба коммитят` — тогда секреты из .env утекут.
- `Оба игнорируют` — но без .env.example коллеги не узнают, какие переменные нужны.
- `.env коммитят, .env.example нет` — наоборот, это опасно.

### Урок 14.5 — Конфликты
**Экраны:** теория → квиз

#### 📖 Теория
**Конфликт** возникает, когда двое поменяли одну и ту же строку по-разному, и Git не может решить сам, чей вариант оставить. Он помечает спорное место:

```text
<<<<<<< твоя версия
amount = 500
=======
amount = 1000
>>>>>>> их версия
```

Решить конфликт — значит выбрать правильный вариант (или объединить), убрать метки `<<<`, `===`, `>>>` и закоммитить. Конфликты — это норма командной работы, не повод паниковать.

🐍 Пай: «Конфликт — не катастрофа, а вопрос Git: „чью строку оставить?“. Ты отвечаешь, убираешь метки, коммитишь».

#### ❓ Квиз
Когда в Git возникает конфликт слияния?
- ✅ `Когда два человека по-разному изменили одну и ту же строку`
- `Когда код не запускается` — это не про Git, а про ошибку выполнения.
- `Когда забыли commit message` — это просто не даст закоммитить, но не конфликт.
- `Всегда при merge` — нет, чаще всего слияние проходит без конфликтов.
